# Скилл Team-Lead — актуальный план

Цель: при запуске `/team-lead` гарантированно собрать локальную команду из 4 pi-сессий в одном cmux workspace и одной pi-link сети: `<ir>@team-lead`, `<ir>@researcher`, `<ir>@coder`, `<ir>@reviewer`.

## Что считаем источником истины

- `<ir>` — это basename текущей директории.
- Текущая сессия — всегда `team-lead` для этого `<ir>`.
- Источник факта о подключённых ролях — вывод `/link` в текущей сессии.
- Создаём только `surface` в текущем cmux workspace. Новый workspace создавать нельзя.
- Недостающие дочерние сессии запускаются через `pi --link`, затем явно переименовываются командой `/link-name <ir>@<role>`.

## Почему старый план ломается

1. Использовался неверный примитив cmux: нам нужен `new-surface`, а не `new-workspace` и не `new-pane` как основной механизм.
2. Команды отправлялись слишком рано: shell или сам `pi` ещё не готовы, поэтому `send`/`send-key` уходят в пустоту или не туда.
3. Автоматизация опиралась на fixed sleep, хотя запуск `pi --link` может занимать 10–20 секунд и более.
4. Нельзя слепо доверять `surface:<n>` ref в автоматизации cmux: в известных багах `send`/`send-key`/`read-screen` с surface ref могут работать некорректно или молча фолбечиться на focused surface. Для адресации нужно резолвить UUID поверхности.
5. `link_list` не является каноническим интерфейсом для этого расширения. Канонический пользовательский интерфейс pi-link — `/link`, `/link-connect`, `/link-name`.

## Документированные ограничения, которые надо учесть

### pi-link

- `/link-connect` работает и без `--link`, но `pi --link` всё равно нужен для автоподключения дочерних сессий на старте.
- `/link` показывает name, role, online count, agent status и cwd для терминалов — этого достаточно, чтобы понять, какие роли уже существуют в текущем проекте.
- `/link-name <name>` сохраняет preferred name в сессии. Если имя занято, hub может выдать суффикс `-2`, `-3`, поэтому после rename нужно верифицировать фактическое имя через `/link`.

### cmux

- `cmux new-surface` создаёт новую surface в текущем или указанном pane/workspace; это именно то, что нам нужно.
- Для programmatic automation нельзя опираться только на факт создания surface: перед отправкой команд надо проверять готовность surface.
- Для readiness и адресации нужно использовать комбинацию `list-surfaces --json --id-format both`, `surface-health`, при необходимости `read-screen`.
- Если нужен таргетинг конкретной поверхности, использовать её UUID, а не `surface:<n>` ref.

### pi / pi-mono

- Скилл должен жить в `.agents/skills/.../SKILL.md` или `~/.agents/skills/.../SKILL.md`; это соответствует текущему размещению.
- `/team-lead` — это entrypoint пользовательского флоу. Сам скилл даёт инструкции модели, а не заменяет runtime-логику pi.

## Целевой флоу

### Шаг 1. Нормализовать текущую сессию

1. Получить `<ir>` как basename текущей директории.
2. В текущей сессии выполнить `/link-connect`.
3. В текущей сессии выполнить `/link-name <ir>@team-lead`.
4. В текущей сессии выполнить `/link` и зафиксировать фактический roster только для terminal'ов с cwd в текущем проекте.

### Шаг 2. Определить, каких ролей не хватает

Обязательный набор ролей:

- `<ir>@team-lead`
- `<ir>@researcher`
- `<ir>@coder`
- `<ir>@reviewer`

Если какая-то роль уже есть в `/link` для текущего cwd, её не пересоздаём.

### Шаг 3. Получить текущий cmux workspace

Источник workspace в порядке приоритета:

1. `CMUX_WORKSPACE_ID`
2. `cmux current-workspace`

Работа всегда ведётся в этом workspace. Новый workspace не создаётся.

### Шаг 4. Для каждой недостающей роли создать отдельную surface

Для каждой роли из списка missing roles:

1. Создать новую terminal surface в текущем workspace.
2. Немедленно резолвить её UUID через `cmux list-surfaces --workspace <ws> --json --id-format both`.
3. Дождаться, пока `surface-health` для этой surface покажет рабочее состояние. Минимум — surface существует и `in_window=true`.
4. Переименовать surface/tab в `<ir>@<role>`.

Важно: создавать именно отдельную `surface`, а не новый workspace и не полагаться на случайно активную pane.

### Шаг 5. Запустить в surface новый pi

В каждую созданную surface отправить:

```bash
pi --link
```

Но не через blind sleep. Нужна стратегия ожидания:

1. После создания surface ждать готовность shell polling-ом, а не фиксированной паузой.
2. После отправки `pi --link` polling-ом читать экран surface до появления признаков, что pi уже запустился и принимает ввод.
3. Только после этого отправлять `/link-name`.

Разрешённые признаки готовности `pi`:

- на экране больше нет обычного shell prompt и уже виден интерфейс pi;
- `read-screen` стабильно возвращает контент самой pi-сессии;
- повторная проверка через короткий интервал подтверждает, что surface жива и ввод не теряется.

Точный детектор можно уточнить при реализации, но он обязан быть polling-based с timeout и диагностикой, а не `sleep(20)`.

### Шаг 6. Назначить имя роли в дочерней сессии

После подтверждённого запуска `pi` отправить в соответствующую surface:

```bash
/link-name <ir>@<role>
```

Затем отправить Enter и проверить из текущей team-lead сессии через `/link`, что в roster появилось нужное имя для текущего cwd.

Примечание: в дочерних сессиях отдельный `/link-connect` не обязателен, если они стартовали через `pi --link` и уже подключились автоматически. Если по факту не появились в `/link`, тогда допустим fallback: отправить `/link-connect`, дождаться подключения и повторить `/link-name`.

## Что должно быть вынесено в Python helper

Да, orchestration создания surface и ожидания готовности стоит вынести в Python-скрипт.

В Python нужно оставить то, что плохо и хрупко делать голым bash:

- вызовы cmux с retry и разбором JSON;
- поиск freshly created surface и резолв её UUID;
- polling `surface-health` / `read-screen` с timeout;
- отправка текста + Enter в нужную surface;
- ожидание старта `pi --link` по наблюдаемым признакам;
- переименование surface/tab;
- возврат структурированного результата: role, workspace id, surface ref, surface uuid, итоговый status.

В самом skill остаётся orchestration верхнего уровня:

- вычислить `<ir>`;
- привести текущую сессию к `<ir>@team-lead>`;
- проверить roster через `/link`;
- вызвать Python helper для недостающих ролей;
- после каждого helper-run перепроверить roster через `/link`.

## Требования к helper-скрипту

Скрипт должен принимать минимум:

```bash
python3 spawn-agent.py <role> <ir> [workspace-id]
```

Скрипт обязан:

1. Работать в текущем cmux workspace, не создавая новый workspace.
2. Создавать новую terminal surface.
3. Находить UUID созданной surface.
4. Дожидаться готовности surface к вводу.
5. Запускать `pi --link`.
6. Дожидаться готовности pi принимать slash-команды.
7. Отправлять `/link-name <ir>@<role>`.
8. По необходимости отправлять `/link-connect` как fallback, а не как основной путь.
9. Переименовывать surface/tab в `<ir>@<role>`.
10. Завершаться ошибкой, если readiness не достигнут за timeout, а не молча продолжать.

## Failure handling

- Если cmux недоступен — прекратить флоу и явно сообщить об этом.
- Если не удалось определить workspace — прекратить флоу, не подставлять `workspace:1` молча.
- Если surface создана, но не стала ready — закрыть её или пометить как failed и не слать команды в неизвестное состояние.
- Если `/link-name` привёл к имени с суффиксом (`-2` и т.д.), это считать конфликтом и явно показывать пользователю.
- Если в `/link` появился terminal из другого cwd, игнорировать его при расчёте missing roles.

## Критерии готовности реализации

После запуска `/team-lead` в проекте с basename `<ir>` в активной pi-link сети и текущем cmux workspace должны существовать ровно эти роли для текущего cwd:

- `<ir>@team-lead` — текущая сессия
- `<ir>@researcher`
- `<ir>@coder`
- `<ir>@reviewer`

Для каждой дочерней роли должна существовать отдельная cmux surface с именем `<ir>@<role>`, внутри которой реально запущен `pi --link`, а не просто открыт shell.

## Что меняем по сравнению со старым SPEC

- Убираем упор на `new-pane`; основной примитив теперь `new-surface`.
- Убираем fixed sleep как основной механизм ожидания.
- Убираем зависимость от `link_list`; используем `/link` как канонический источник статуса.
- Переносим низкоуровневую cmux-автоматику в Python helper.
- Добавляем обязательную верификацию roster после каждого шага через текущую team-lead сессию.
